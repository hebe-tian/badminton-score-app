# 一、需求概述

本次需求为新增比赛结束后的得分记录弹窗功能，核心要求：在不修改原有任何功能的前提下，比赛结束后自动弹出得分记录弹窗；弹窗长度固定，至少展示8行得分数据，支持弹窗列表滑动；得分记录按指定表格格式展示，且根据比赛模式（单打、双打、五羽伦比）差异化展示“得分者”字段，确保功能新增不影响原有流程，便于agent准确实现。

# 二、核心功能需求（不修改原有功能）

## 2\.1 弹窗基础要求

- 触发时机：比赛正式结束（某一方达到胜利条件、用户手动结束比赛等原有结束场景）后，自动弹出得分记录弹窗，无需用户额外操作。

- 弹窗样式：弹窗长度固定（具体尺寸无需自定义，保证视觉统一即可），弹窗内得分记录列表支持垂直滑动。

- 数据展示：弹窗内得分记录至少展示8行；若比赛实际得分次数不足8行，需用空行填充（空行仅保留表格框架，无数据），确保满足“至少8行”要求；若得分次数超过8行，通过滑动列表查看全部记录。

- 弹窗操作：弹窗需提供关闭按钮（如“确定”），用户点击关闭后弹窗消失，返回原有比赛结束后的界面，不新增其他操作逻辑，不影响原有结束流程。

## 2\.2 得分记录表格格式

得分记录以表格形式展示，固定3列，列名及格式如下：

|序号|得分者|队伍分数|
|---|---|---|
|示例1|（根据模式展示对应内容）|A队：X分 \| B队：Y分|
|示例2|（根据模式展示对应内容）|A队：X分 \| B队：Y分|

补充说明：

- 序号：按得分先后顺序依次编号（1、2、3\.\.\.），从比赛第一次得分开始计数，直至比赛结束。

- 队伍分数：展示该次得分后，双方队伍的实时总分数，格式固定为“A队：X分 \| B队：Y分”（X、Y为对应队伍当前总得分）。

## 2\.3 不同比赛模式下“得分者”展示规则

根据当前比赛模式（单打、双打、五羽伦比），差异化展示“得分者”字段内容，严格遵循以下规则，不新增其他展示形式：

- 单打模式：得分者展示为“得分选手”（即该次得分的单个选手姓名，与原有选手名称同步，如“张三”）。

- 双打模式：得分者展示为“选手A，选手B”（即该次得分的两名选手姓名，用英文逗号分隔，与原有选手名称同步，如“张三，李四”）。

- 五羽伦比模式：得分者展示为“当前在场的两位选手”（即该次得分时，场上正在参与比赛的两名选手姓名，用英文逗号分隔，与当前场上选手同步，如“A3，B3”）。

# 三、交互说明

- 弹窗弹出：比赛结束后自动弹出，无需用户触发，弹窗层级高于原有界面，不遮挡原有界面核心内容（仅展示得分记录，不干扰原有结束后操作）。

- 列表滑动：当得分记录超过8行时，弹窗内列表支持垂直滑动，滑动过程中表格列名（序号、得分者、队伍分数）固定不动，仅内容区域滑动。

- 弹窗关闭：用户点击弹窗内“确定”按钮（或弹窗边缘关闭区域），弹窗立即消失，返回原有比赛结束后的界面，无其他后续操作，不影响原有功能流程。

# 四、关键约束

- 不修改原有任何功能：包括但不限于比赛规则、计分逻辑、界面布局、原有弹窗（若有）等，仅新增本次需求的得分记录弹窗。

- 数据同步：得分记录中的选手姓名、队伍分数，需与比赛过程中记录的实际数据完全同步，与原有功能中选手名称、计分结果保持一致，不出现数据偏差。

- 兼容性：弹窗功能需适配原有所有比赛模式（单打、双打、五羽伦比），无需额外修改模式相关代码，仅根据模式判断“得分者”展示内容。

# 五、技术实现规范

## 5.1 数据结构扩展
为了支持弹窗展示及未来的导出功能，需对 `ScoreHistoryEntry` 进行增强：
- **基础信息**：序号 (`sequenceNumber`)、双方实时比分 (`teamAScore`, `teamBScore`)。
- **得分者详情**：增加 `scorers: string[]` 字段，记录该回合得分的具体人员姓名。
- **场上阵容快照**：针对五羽伦比模式，需在加分瞬间保存当时的 `teamACourtPositions` 和 `teamBCourtPositions`，确保“得分者”能准确反映当时在场的两位选手。

## 5.2 历史记录追踪逻辑
- **实时同步**：在 `App.tsx` 的加分处理函数（如 `handleSinglesScore`）中，必须同步调用历史记录更新逻辑，确保内存中的 `history` 数组与比赛进程完全一致。
- **非持久化**：历史记录仅作为本次比赛的一次性回顾，存储在组件状态中，不写入本地存储。

## 5.3 UI 渲染细节
- **空行填充策略**：当实际得分记录不足 8 行时，使用虚拟数据补齐至 8 行。空行单元格内不得包含任何文字或占位符，保持视觉上的完全留白。
- **样式一致性**：弹窗组件应复用现有项目的 Tailwind CSS 风格（如 `rounded-3xl`, `bg-opacity-90`），并根据当前比赛模式动态适配主题色。
- **可扩展性预留**：弹窗内容区域应包裹在具有唯一标识的容器中，便于未来通过 `html2canvas` 等工具直接捕获 DOM 节点生成图片。

# 六、备注

1\.  弹窗长度固定，无需用户自定义调整，确保不同设备上展示效果一致；

2\.  空行填充要求：若比赛得分不足8行，空行需保留表格的边框和列名对应位置，仅单元格内无任何文字；

3\.  agent需重点关注“不修改原有功能”的约束，仅新增弹窗及相关逻辑，确保新增功能与原有功能无冲突、无干扰；

4\.  得分记录的顺序严格按照得分先后排列，不可颠倒，序号连续无缺失。
