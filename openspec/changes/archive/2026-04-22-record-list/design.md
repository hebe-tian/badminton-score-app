## Context

当前羽毛球计分应用（badminton-score-app）已具备单打、双打和五羽伦比三种模式的完整计分逻辑。然而，比赛结束后用户只能看到最终比分，缺乏对每一分得分过程及当时场上人员配置的详细回顾。现有的 `ScoreHistory` 结构虽然定义了历史记录接口，但在 `App.tsx` 的实际加分逻辑中并未被有效利用。

## Goals / Non-Goals

**Goals:**
- 实现一个在比赛结束时自动触发的得分记录弹窗。
- 确保弹窗能根据不同模式准确展示“得分者”信息（特别是五羽伦比的实时阵容）。
- 保持 UI 风格与现有应用高度一致，并提供至少 8 行的固定展示区域。
- 为未来可能的“导出图片”或“分享消息”功能预留数据结构支持。

**Non-Goals:**
- 不涉及历史记录的持久化存储（如 LocalStorage 或后端同步）。
- 不修改现有的比赛规则、计分算法或发球轮换逻辑。
- 不包含复杂的动画效果，仅实现基础的弹窗显隐。

## Decisions

1. **数据追踪策略：在 App.tsx 集中维护**
   - **决策**：不在各个 `Match` 组件内部维护历史记录，而是在 `App.tsx` 的顶层状态中统一管理。
   - **理由**：`App.tsx` 是所有加分动作的入口，在此处拦截并记录数据可以确保逻辑的唯一性和准确性，避免在各组件中重复编写记录代码。

2. **五羽伦比阵容快照：扩展 ScoreHistoryEntry**
   - **决策**：在 `ScoreHistoryEntry` 中增加 `onCourtPlayers` 字段，用于存储得分瞬间的 `teamACourtPositions` 和 `teamBCourtPositions`。
   - **理由**：五羽伦比的场上人员是动态变化的，只有保存当时的“快照”，才能在赛后准确还原出“谁和谁在场”。

3. **UI 实现：通用 Modal 组件 + 差异化渲染**
   - **决策**：创建一个独立的 `ScoreRecordModal.tsx` 组件，通过 `props` 接收模式和数据进行渲染。
   - **理由**：复用性高，且便于统一控制弹窗样式（如固定高度、滚动条、空行填充逻辑）。

4. **空行处理：虚拟数据填充**
   - **决策**：在渲染前判断数组长度，若不足 8 则补充空对象。
   - **理由**：简化表格渲染逻辑，确保视觉上的整齐划一。

## Risks / Trade-offs

- **[风险] 内存占用**：虽然单次比赛数据量很小，但若用户长时间不刷新页面并进行多场比赛，内存中的历史记录会累积。
  - **缓解**：在每次点击“重新开始”或“返回主页”时清空对应的历史记录状态。
- **[风险] 五羽伦比数据复杂性**：需要在加分瞬间精确捕获场上位置。
  - **缓解**：在 `handleWuYunLunBiScore` 执行完加分逻辑后，立即从最新的 `state` 中提取位置信息并存入历史记录。
